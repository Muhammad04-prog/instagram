# Что нужно починить на бэкенде

Один документ, по которому фронт можно довести до 100%. Всё ниже — факты, снятые с
`https://backend-instagram-kvv4.onrender.com` и с вашего же Swagger
(`/api/docs-json`), а не пожелания.

**Где мы:** фронт (Next.js 16) переведён на новый API, **147 из 170** endpoint'ов
вызываются из живого UI. Из оставшихся 23:

- **19** — недоделки фронта (музыка, Spotify, тренды, репосты, редактирование
  сообщения и т.д.). Это наша работа, к вам вопросов нет, доделаем;
- **1** (`GET /posts`) — не подключён сознательно: `/search/explore` отдаёт ровно ту
  же сетку, дублировать незачем;
- **3** — упираются в вас: `/chats/{id}/call`, `/live/requests/{id}/accept`,
  `/live/requests/{id}/decline`.

Но главная проблема не в этих трёх. **БД лежит всё время миграции, поэтому ни один
из 147 не подтверждён живым ответом.** Всё, что мы проверили — сборкой, типами и
Playwright'ом на заглушках. Это ловит ошибки фронта, но не расхождения со Swagger.
На прошлом бэкенде именно живая проверка ловила такие расхождения в 8 фазах из 10
(например, `gender` читался строкой `"Male"`, а на запись принимал `0|1`).

## 1. 🔴 БД / Redis / MinIO лежат — это блокер №1

```
GET /api/health
{"data":{"status":"degraded","database":"down","redis":"down","storage":"down","uptimeSec":1241}}
```

**Само NestJS-приложение живо.** Упал только доступ к внешним сервисам, и вот
доказательства — обратите внимание, что коды ответов разные, и это диагностика:

| Запрос                                      | Ответ                     | Вывод                           |
| ------------------------------------------- | ------------------------- | ------------------------------- |
| `POST /api/auth/register` с пустым телом    | **400** + ошибки по полям | `class-validator` работает      |
| `GET /api/posts`, `GET /api/music/trending` | **401** Unauthorized      | Guard'ы работают (они до БД)    |
| `POST /api/auth/check-username`             | **500** `INTERNAL_ERROR`  | Любое обращение **к БД** падает |
| `POST /api/auth/login`                      | **500** `INTERNAL_ERROR`  | То же                           |

То есть 400 и 401 отвечают штатно, а всё, что идёт в БД, — 500. Падает не код,
падает подключение.

**Наша гипотеза: в прод уехал конфиг локальной разработки.** В вашем же Swagger
примеры такие:

```
MusicDto.coverUrl      → "http://localhost:9000/instagram/music/covers/x.webp"    ← MinIO
PostMusicDto.streamUrl → "http://localhost:3000/api/music/35/stream"
LiveTokenDto.wsUrl     → "ws://localhost:7880"                                     ← LiveKit
```

Внутри Render `localhost:9000` не существует. То, что **все три** сервиса
(`database`, `redis`, `storage`) отвалились разом, ложится ровно в эту версию:
env указывает на docker-compose разработчика.

**Что сделать:**

1. Открыть логи сервиса на Render — там настоящая ошибка Prisma. Наружу она не
   видна, клиент получает обезличенное `"Internal server error"`.
2. Проверить на Render: `DATABASE_URL`, `REDIS_URL`, endpoint MinIO.
3. Если Postgres на бесплатном тарифе (Neon / Supabase) — он засыпает сам, нужен
   wake или платный план.

**Как проверить, что готово:** `GET /api/health` → `"status":"ok"` и все три в `up`.
Затем `POST /api/auth/check-username` должен отвечать 200/409, а не 500.

## 2. 🔴 Публичный хост медиа

Фронт обязан знать домен, с которого приходят картинки: `next/image` не грузит хост,
которого нет в `next.config.ts → images.remotePatterns`. Это защита фреймворка, а не
наш каприз — иначе любой чужой URL в БД превратился бы в загрузку с чужого сервера.

Ни одного настоящего URL медиа мы не видели — только `localhost:9000` в примерах.

**Что нужно:** публичный домен MinIO/S3, и чтобы `avatarUrl`, `media[].url`,
`coverUrl`, `streamUrl` отдавались с него, а не с `localhost`. Пришлите домен —
мы добавим его в `remotePatterns`.

---

## 3. 🔴 Сокет: есть, но подключиться нечем

Хендшейк `/socket.io/` на корне домена отвечает — значит, gateway поднят. Но
пользоваться им нельзя по двум причинам:

**3.1. Нечем авторизоваться.** Access-токен у нас лежит в **httpOnly-cookie** (так
и должно быть — из JS он недоступен, это защита от XSS). Сокет идёт с браузера
напрямую на ваш домен, а это cross-origin: кука туда не уедет, и достать её из JS,
чтобы положить в `auth`, мы не можем и не будем.

**Что нужно** — любое одно:

- `POST /socket/ticket` → `{ ticket: "<одноразовый, 30 сек>" }`, который фронт
  получает обычным HTTP (там кука есть) и передаёт в
  `io(url, { auth: { ticket } })`; или
- разрешить `withCredentials` + CORS с нашим origin, чтобы кука доехала сама.

**3.2. Событий нет в документации.** Swagger описывает только HTTP. Ни одного имени
события (`message:new`, `typing`, `live:comment`, …), ни payload'ов. Угадывать
имена мы не будем — это гарантированно молчащий сокет.

**Что нужно:** список событий с payload'ами. Хотя бы так:

```
message:new      → MessageDto
message:read     → { chatId, messageId, userId }
typing           → { chatId, userId, isTyping }
live:comment     → LiveCommentDto
live:like        → { liveId, likesCount }
live:viewers     → { liveId, viewersCount }
live:join-request→ JoinRequestDto        ← закрыло бы и пункт 6
```

Пока этого нет: чат работает на polling'е, live-чат показывает только свои
комментарии, звонки (`/chats/{id}/call`) не подключены вообще.

## 4. 🔴 LiveKit: `wsUrl` указывает на машину зрителя

`POST /live/start`, `/live/{id}/join` и `/live/requests/{id}/accept` возвращают
`LiveTokenDto` с `wsUrl: "ws://localhost:7880"`.

Для браузера `localhost` — это **компьютер самого зрителя**, а не ваш сервер.
Никакое видео оттуда прийти не может: ни с LiveKit SDK, ни без него, ни при каких
настройках фронта.

**Что нужно:**

1. Публичный `wss://`-хост LiveKit (например `wss://live.instagram-api.softclub.tj`),
   и чтобы `wsUrl` приходил именно им;
2. подтвердите, что `token` — обычный LiveKit access token (тогда фронту хватит
   `livekit-client`);
3. `ws://` не подойдёт: страница по https, браузер заблокирует незащищённый сокет.

**Что сейчас:** экран эфира показывает всё состояние (зрители, лайки, комментарии,
камера/звук, статистика) и честно пишет «видео не подключено». Подставной картинки
мы не рисуем.

## 5. 🔴 Комментарии в эфире можно отправить, но нельзя прочитать

Есть `POST /live/{id}/comment` → возвращает **только что созданный** комментарий.
`GET /live/{id}/comments` **не существует**.

То есть API умеет только писать. Поток чужих комментариев получить нечем, а сокетом
это не закрыть (см. §3). Live-чат без чужих комментариев — не live-чат.

**Что нужно:** `GET /live/{id}/comments?cursor=&limit=` → `LiveCommentDto[]`
(как у постов). Даже без сокета этого хватит — будем опрашивать раз в 2–3 секунды.

**Что сейчас:** зритель видит только свои комментарии, и экран об этом прямо пишет.

---

## 6. 🔴 Хост не может принять заявку — id заявки ему негде взять

`POST /live/requests/{id}/accept` и `/decline` принимают id заявки. Но проследите
путь этого id — он никуда не ведёт:

- эндпоинта «список заявок» нет — `GET /live/{id}/requests` отсутствует;
- `JoinRequestDto` с id возвращается **тому, кто попросился**, а не хосту;
- `NotificationDto` несёт `postId`, `commentId`, `storyId`, `noteId` — и **не несёт**
  `requestId`. У типов `LIVE_JOIN_REQUEST`, `LIVE_JOIN_ACCEPTED`, `LIVE_JOIN_DECLINED`
  нет даже `liveId`.

Итого id не доходит до хоста ни одним путём. **Это единственные 2 из 18 live-endpoint'ов,
что не подключены** — и хук под них мы не писали: хук без экрана только делает
отчётность красивее, чем приложение.

**Что нужно** (лучше оба):

1. `GET /live/{id}/requests` → `JoinRequestDto[]` — у хоста появится панель заявок;
2. добавить в `NotificationDto` поля `liveId?: string` и `requestId?: number`.

Второе заодно чинит `LIVE_STARTED`: сейчас по этому уведомлению **некуда перейти**,
id эфира в нём нет.

## 7. ⚠️ Нет «постов по локации»

Место у поста есть (`PostDto.location`), обратного запроса — нет. `/search?q=` отдаёт
сами локации, но не их ленту.

`/explore/locations/{id}` сейчас показывает только само место и честно пишет, что
ленты нет. В IG это одна из основных страниц.

**Что нужно:** `GET /locations/{id}/posts?cursor=&limit=` → `PostDto[]`
(как `/search/hashtag/{name}`).

---

## 8. ⚠️ Курсор не работает там, где у DTO нет `id`

Курсор у вас — «id последнего элемента предыдущей страницы». Но у этих DTO своего
`id` нет, поэтому «показать ещё» физически нечем:

- `StoryViewerDto` (`GET /stories/{id}/viewers`)
- `ActivityItemDto` (`GET /profile/me/activity`)
- `ProfileViewDto` (`GET /notifications/profile-views`)

Сейчас фронт берёт разом `limit=50`, дальше список просто обрезается.

**Что нужно:** добавить в эти DTO `id` — или отдавать `nextCursor` в ответе.

---

## 9. ⚠️ Нет поиска пользователя по точному `userName`

`@упоминание` в подписи ведёт на `/u/{userName}`, а роуты профиля — на uuid. Сейчас
фронт ищет через `/users?q=` и **сам выбирает точное совпадение**, потому что
подстрочный поиск матчит и `fullName` (`er` → `eraj`, но и `america`).

**Что нужно:** `GET /users/by-username/{userName}` → `UserBriefDto`. Уберём костыль.

---

## 10. ⚠️ `GET /auth/me` — неясно, что приходит

Описан как «текущий пользователь + профиль», но по схеме не видно, что именно:
голый `AuthUserDto` или конверт `{ user, profile }`. Фронт сейчас принимает **оба**
варианта — уточните, уберём лишнюю ветку.

---

## 11. ⚠️ Коллекции сохранений односторонние

`POST /posts/{id}/favorite` принимает `collection` (имя), но endpoint'а «список моих
коллекций» нет — прочитать их обратно нельзя. Поэтому в UI ввод имени руками, а не
выбор из списка, как в IG.

**Что нужно:** `GET /profile/me/collections` → `{ name, postsCount, coverUrl }[]`.

---

## 12. ⚠️ Уведомление не несёт превью поста

В `NotificationDto` есть `postId`, но нет картинки. В IG в строке уведомления справа
стоит миниатюра поста. Сейчас её нет — выдумывать не стали.

**Что нужно:** `postThumbUrl?: string` в `NotificationDto`.

---

## Приёмка

Порядок важен: без §1 остальное проверить нельзя.

1. `GET /api/health` → `status: ok`, `database`/`redis`/`storage` — все `up`.
2. `POST /api/auth/register` → `login` → `GET /auth/me` отвечают 200. **Пришлите тестовый
   аккаунт** — прогоним регистрация → лента → пост → чат и сверим каждый DTO с живым
   ответом.
3. Любой `avatarUrl` / `media[].url` открывается в браузере с публичного домена (не
   `localhost`).
4. `LiveTokenDto.wsUrl` начинается с `wss://` и указывает на живой LiveKit.
5. Сокет: есть способ авторизации (ticket или CORS+cookie) и список событий.
6. `GET /live/{id}/comments` отвечает массивом.
7. `NotificationDto` несёт `liveId` + `requestId` — либо есть `GET /live/{id}/requests`.

Как только §1 закрыт — мы прогоняем все 147 endpoint'ов вживую и присылаем список
расхождений со Swagger (если они будут). Пункты §7–12 не блокируют, но каждый из них
сейчас — это место, где в UI стоит заглушка вместо нормального экрана IG.
