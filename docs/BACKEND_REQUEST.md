Но проверить хоть один ответ живьём **невозможно**: БД лежит. Ниже — только факты, снятые с

## 1. 🔴 Главное: БД / Redis / MinIO лежат

```
GET /api/health
{"data":{"status":"degraded","database":"down","redis":"down","storage":"down","uptimeSec":1241}}
```

**Само приложение NestJS живо** — упал только доступ к внешним сервисам. Доказательства:

| Запрос                                      | Ответ                            | Что это значит                     |
| ------------------------------------------- | -------------------------------- | ---------------------------------- |
| `POST /api/auth/register` с пустым телом    | **400** + подробные ошибки полей | Валидация (`class-validator`) жива |
| `GET /api/posts`, `GET /api/music/trending` | **401** Unauthorized             | Guard'ы отрабатывают (до БД)       |
| `POST /api/auth/check-username`             | **500** `INTERNAL_ERROR`         | Любое обращение **к БД** падает    |
| `POST /api/auth/login`                      | **500** `INTERNAL_ERROR`         | То же                              |

**Что нужно сделать:**

1. Открыть логи сервиса на Render — там будет настоящая ошибка Prisma (неверный `DATABASE_URL`,
   БД suspended, лимит подключений, IP-allowlist). Наружу она не видна: клиент получает только
   обезличенное `"Internal server error"`.
2. Проверить переменные окружения на Render: `DATABASE_URL`, `REDIS_URL`, endpoint MinIO.
3. Если Postgres на бесплатном тарифе (Neon / Supabase) — он засыпает сам; нужен wake или платный план.

**Подозрение — конфиг от локальной разработки уехал в прод.** В самом Swagger примеры такие:

```
MusicDto.coverUrl      → "http://localhost:9000/instagram/music/covers/x.webp"   ← MinIO
PostMusicDto.streamUrl → "http://localhost:3000/api/music/35/stream"
```

`localhost:9000` внутри Render не существует. То, что **все три** сервиса (`database`, `redis`,
`storage`) упали разом, укладывается ровно в это: env указывает на docker-compose разработчика.

## 3. Мелочи, которые всплыли по Swagger

Ни одна не блокирует — но лучше поправить, чем оставить.

1. **Курсор не пагинируется там, где у DTO нет `id`.** Курсор — «id последнего элемента предыдущей
   страницы», а у этих DTO своего id нет, поэтому «показать ещё» физически нечем:
   - `StoryViewerDto` (`GET /stories/{id}/viewers`) — берём разом лимит 50, дальше обрежется;
   - `ActivityItemDto` (`GET /profile/me/activity`);
   - `ProfileViewDto` (`GET /notifications/profile-views`).

   Достаточно добавить в эти DTO `id` (или отдавать `nextCursor` в ответе).

2. **Нет поиска пользователя по точному `userName`.** `@упоминание` в подписи ведёт на `/u/{userName}`,
   а роуты профиля — на uuid. Сейчас фронт ищет через `/users?q=` и **сам выбирает точное совпадение**
   (подстрочный поиск матчит и `fullName`: `er` → `eraj`, `america`). Один
   `GET /users/by-username/{userName}` убрал бы этот костыль.

3. **`GET /auth/me` описан как «текущий пользователь + профиль»**, но по схеме не видно, что именно
   приходит: голый `AuthUserDto` или конверт `{ user, profile }`. Фронт сейчас принимает **оба**
   варианта. Уточните — уберём ветку.

4. **Коллекции сохранений односторонние.** `POST /posts/{id}/favorite` принимает `collection` (имя),
   но endpoint'а «список моих коллекций» нет — прочитать их обратно нельзя. Поэтому в UI ввод имени,
   а не выбор из списка.

5. **Уведомление не несёт превью поста.** В `NotificationDto` есть `postId`, но нет картинки, —
   в IG в строке справа стоит миниатюра. Сейчас её нет (выдумывать не стали). Пригодилось бы
   `postThumbUrl`.

## 6. 🔴 Видео и часть картинок не грузятся — Cloudinary `fetch/` отдаёт 404 (найдено 19.07.2026, БД уже жива)

Часть `media[].url` / `avatarUrl` приходят завёрнутыми в Cloudinary remote-fetch:

```
https://res.cloudinary.com/mnxhfazc/video/upload/v1/fetch/https://instagram-api.softclub.tj/images/<id>.mp4
```

Каждый такой URL отвечает **404** с `x-cld-error: Resource not found - fetch/...`. Проверено на
5 реальных reel-видео с `/api/posts/reels` — 4 из 5 были завёрнуты в `fetch/` и все 4 дали 404.

**Это не «старый хост умер».** Прямой запрос к тому же самому файлу на старом хосте, в обход
Cloudinary, отвечает нормально:

```
curl -I https://instagram-api.softclub.tj/images/16e38c16-960a-42a4-8e99-cd757344db0f.mp4
→ HTTP/1.1 206 Partial Content, Content-Type: video/mp4, Content-Length: 3243329
```

Пятое видео в той же выборке пришло вообще без обёртки Cloudinary (голый URL старого хоста) и
играет нормально — то есть проблема не в файле и не в хосте, а именно в Cloudinary `fetch/`
delivery: он 404-ит **для любого** origin URL, включая тот, что только что подтверждённо жив
(проверено `curl` напрямую на res.cloudinary.com). Похоже на отключённый в настройках Cloudinary
аккаунта (`mnxhfazc`) `Allowed fetch domains` / "Enable fetch" — эта опция по умолчанию выключена
у новых аккаунтов как защита от абьюза.

**Что нужно сделать (любое из двух):**

1. В Cloudinary Console → Settings → Security включить fetch remote media для нужных доменов
   (`instagram-api.softclub.tj` и что бы ни было текущим хранилищем), **или**
2. Перестать заворачивать URL в `fetch/` и отдавать голый origin URL — судя по пятому видео из
   выборки, он и так грузится напрямую.

Фронт починить это не может: он просто рендерит тот URL, который прислал бэкенд. Со своей стороны
добавлена деградация — сломанное видео/картинка показывает плейсхолдер вместо чёрного экрана —
но сами медиа так не заиграют.

**Тот же симптом и на свежей загрузке, не только на `fetch/`.** Голосовое сообщение, отправленное
20.07.2026 (`POST /chats/{id}/messages` с mp3), в ответе получило нормальный
`mediaUrl` (`res.cloudinary.com/mnxhfazc/raw/upload/v1/audio/...mp3`), но сам файл сразу же отвечает
404 с тем же `x-cld-error: Resource not found`:

```
curl -I https://res.cloudinary.com/mnxhfazc/raw/upload/v1/audio/2026/07/1d3d525a-...-mp3
→ HTTP/1.1 404, x-cld-error: Resource not found - audio/2026/07/1d3d525a-...-mp3
```

Это уже не `fetch/`-обёртка, а собственная загрузка (`raw/upload`) — файл, который бэкенд сам только
что положил в Cloudinary, тут же не находится. Похоже на рассинхрон `resource_type` (загружено как
`raw`, а читается/линкуется под другим типом) или на саму учётку Cloudinary — стоит проверить оба
случая вместе.

## 7. ⚠️ Звонки: TURN не настроен — `hasTurn: false` (найдено 20.07.2026)

```
GET /api/chats/calls/ice-servers
→ {"iceServers":[{"urls":["stun:stun.l.google.com:19302"]}],"hasTurn":false}
```

Только публичный STUN Google, TURN отсутствует — то, что уже честно описано в самом Swagger
(`IceServersDto.hasTurn`: "false — TURN не настроен: звонок между двумя NAT может не соединиться").

Фронт эти `iceServers` уже забирает и отдаёт в `RTCPeerConnection` (звонок технически идёт через
PeerJS, не через наш сокет — см. `usePeerCall`). Автоматический тест с двумя реальными аккаунтами
на одной машине/сети прошёл: аудио-трек `live`, `currentTime` идёт, звук действительно играет.
**Но без TURN это работает только пока оба пира могут договориться напрямую** — как только один из
двух за симметричным NAT (почти любая мобильная сеть, часть Wi-Fi роутеров), медиа-путь никогда не
установится: экран звонка покажет ONGOING, но звука не будет ни у одной стороны. Это и есть жалоба
«звонок происходит, а потом звука нет».

**Нужно:** `TURN_URLS` / `TURN_USERNAME` / `TURN_PASSWORD` в .env бэкенда (переменные уже
предусмотрены самим эндпойнтом — их просто никто не заполнил). Любой TURN-провайдер подойдёт
(coturn на своём сервере, Twilio, Metered, Xirsys). Фронт ничего менять не должен — как только
`hasTurn` станет `true`, звонки между разными сетями заработают сами.

## 4. 🔴 Сокет: есть, но подключиться нечем

Сокет **живой** — Socket.IO отвечает на handshake:

```
GET https://backend-instagram-a4k6.onrender.com/socket.io/?EIO=4&transport=polling
0{"sid":"yE2e83YGYMDlqj6GAAAA","upgrades":["websocket"],"pingInterval":25000,...}
```

(Он в **корне домена**, не под `/api` — `/api/socket.io/` отдаёт 404.)

Но подключить его мы не можем, и это два разных блокера:

**4.1. Нечем авторизоваться.** Токен у нас живёт **только в httpOnly-cookie** и в клиентский JS не
попадает никогда — это осознанное решение по безопасности (XSS не сможет его украсть). Сокет идёт с
браузера на **другой origin**, поэтому наша cookie туда не уйдёт, а `access_token` мы в JS достать
не можем и не хотим.

Что решило бы (любое из двух):

- endpoint вида `POST /auth/socket-ticket` → короткоживущий одноразовый тикет, который можно отдать
  в `io(url, { auth: { ticket } })`. Тикет не страшно показать JS: он живёт секунды и ничего больше
  не открывает;
- либо сокет принимает cookie: `withCredentials` + `Access-Control-Allow-Credentials: true` +
  `SameSite=None; Secure` на нашей стороне — но это работает только если фронт и сокет на одном
  сайте, что не наш случай.

**4.2. Не описаны события.** В Swagger нет ни одного имени события — только фраза
«WebRTC-сигналинг через сокет». Мы не знаем, что слушать: `message`? `newMessage`? `message:new`?
В какой комнате (`chat:{id}`?), что приходит в payload, как приходит `isOnline` / «печатает» /
входящий звонок. Угадывать имена = писать код, который молча ничего не делает.

**Пришлите список событий** (имя → payload → когда шлётся), хотя бы для:
сообщение создано / отредактировано / удалено, реакция, прочитано, presence (online/last seen),
входящий звонок.

Пока этих двух вещей нет, чат живёт на **опросе раз в 5 секунд**, а кнопку «Позвонить» мы не
делаем: `POST /chats/{id}/call` без сигналинга создаст звонок, который никуда не ведёт.

**Ещё одна утечка localhost:** в Swagger `ws://localhost:7880` — это дефолтный порт LiveKit
(эфиры, Фаза 21). Та же проблема, что и с MinIO из п.1: конфиг локальной разработки уехал в прод.

## 8. 🔴 `POST /posts/{id}/share` с `toStory: true` не создаёт историю (найдено 20.07.2026)

Эндпойнт отвечает честным на вид успехом:

```
POST /posts/16/share  { "toStory": true }
→ 201 { "link": "...", "message": "Публикация отправлена в вашу историю" }
```

Но истории в результате нет. Проверено дважды подряд, на двух разных постах (id 16 и 19):

```
GET /stories/my  → []
GET /stories     → []
```

Само хранилище историй работает — прямая загрузка через `POST /stories` (multipart, обычный путь
создания истории) в том же аккаунте тут же появляется в `GET /stories/my`. Значит проблема именно в
ветке `toStory` у `/posts/{id}/share`: она возвращает успех и, видимо, ссылку на несуществующую
историю, но не пишет саму запись (`StoryDto.fromPostId` для этого явно предусмотрен в схеме, просто
не заполняется).

Фронт вызывает эндпойнт ровно так, как он документирован, получает `201`, инвалидирует кэш
историй — и после этого искренне показывает то, что реально лежит в `/stories/my`, то есть ничего.
Чинить на фронте нечего: сам ответ бэкенда лжёт об успехе.

## 6. Черновики/отложенные посты: read-half есть, write-half нет

`GET /posts/drafts` (мои DRAFT/SCHEDULED) и `PUT /posts/{id}/publish` существуют и подключены на
фронте (`docs/API_MAP_V2.md`, тег `posts`). Но **ничто не может создать** пост со статусом DRAFT
или SCHEDULED в первую очередь:

- `POST /posts` (multipart create) не принимает `status` и не принимает `scheduledAt` — только
  `media`, `caption`, `locationId`, `musicId`, `taggedUserIds`, `filters`, `isReel`;
- `PUT /posts/{id}` (`UpdatePostDto`) принимает только `caption`.

Итог: экран «Черновики» (`/settings/drafts`) честно существует и работает, но **гарантированно
пуст** для любого аккаунта — заполнить его нечем ни одним документированным вызовом.

**Нужно** одно из двух в `POST /posts`:

1. `status?: "DRAFT" | "SCHEDULED"` + `scheduledAt?: string` в теле multipart-запроса создания;
2. либо отдельный `PATCH /posts/{id}/schedule` уже после создания (аналогично `/pin`, `/privacy`).

## 7. 🔴 Live: три дыры, из-за которых эфир нельзя доделать

Все 18 эндпоинтов `live` подключены на фронте, кроме двух — и не потому, что не дошли руки.

### 7.1. `wsUrl` указывает на машину зрителя

`POST /live/start`, `/live/{id}/join` и `/live/requests/{id}/accept` возвращают `LiveTokenDto`
с `wsUrl: "ws://localhost:7880"`. У браузера `localhost` — это **компьютер самого зрителя**, а не
ваш LiveKit. Никакое видео оттуда прийти не может — ни с LiveKit SDK, ни без него.

**Нужно:** публичный `wss://`-хост LiveKit (например `wss://live.instagram-api.softclub.tj`),
и чтобы `wsUrl` приходил именно им. Плюс подтвердите, что токен — обычный LiveKit access token
(тогда фронту хватит `livekit-client`).

Пока этого нет, экран эфира показывает всё состояние (зрители, лайки, комментарии, камера/звук,
статистика) и честно пишет, что видео не подключено. Подставная картинка не рисуется.

### 7.2. Комментарии можно отправить, но нельзя прочитать

Есть `POST /live/{id}/comment` → возвращает **только что созданный** комментарий.
`GET /live/{id}/comments` **не существует**. То есть поток чужих комментариев получить нечем:
API умеет только писать. Сокета, которым это можно было бы закрыть, тоже нет (см. §4).

**Нужен** `GET /live/{id}/comments?cursor=&limit=` → `LiveCommentDto[]` (как у постов).

Сейчас зритель видит только свои собственные комментарии, и экран об этом прямо пишет.

### 7.3. Хост не может принять заявку — id заявки ему негде взять

`POST /live/requests/{id}/accept` и `/decline` принимают id заявки. Но:

- эндпоинта «список заявок» нет — `GET /live/{id}/requests` отсутствует;
- `JoinRequestDto` с id возвращается **тому, кто попросился**, а не хосту;
- `NotificationDto` несёт `postId`, `commentId`, `storyId`, `noteId` — и **не несёт**
  `requestId`. У типов `LIVE_JOIN_REQUEST`, `LIVE_JOIN_ACCEPTED`, `LIVE_JOIN_DECLINED`
  тоже нет `liveId`.

Итого id заявки не доходит до хоста ни одним путём, и split-экран с гостем собрать не из чего.
Эти 2 эндпоинта — единственные из 18, что остались неподключёнными.

**Нужно** любое из двух (лучше оба):

1. `GET /live/{id}/requests` → `JoinRequestDto[]` — тогда у хоста будет панель заявок;
2. добавить в `NotificationDto` поля `liveId?: string` и `requestId?: number`.

Второе заодно чинит `LIVE_STARTED`: сейчас по этому уведомлению **некуда перейти** — id эфира
в нём нет.

## 8. «Просмотр один раз» в чате: та же дыра read/write, что у черновиков

`MessageDto.viewOnce` / `viewOnceOpened` существуют, `POST /chats/messages/{id}/open` подключён и
работает (плейсхолдер «Нажмите, чтобы посмотреть» → реальное медиа один раз). Но **отправить**
такое сообщение нечем:

`POST /chats/{id}/messages` (multipart и JSON — оба варианта) принимает только `text`, `file`,
`replyToId`, `sharedPostId`, `stickerUrl`. Поля `viewOnce` нет ни в одной из двух схем.

**Нужен** `viewOnce?: boolean` в теле `POST /chats/{id}/messages` — тогда composer сможет пометить
вложение как «один просмотр» при отправке, а не только принимать уже помеченные кем-то другим.
