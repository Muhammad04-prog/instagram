# Промпт для Claude Code в репозитории бэкенда

> Отдать целиком, одним сообщением, в Claude Code, запущенный в папке бэкенда
> (NestJS + Prisma). Он самодостаточен: агент не видит фронт и не должен его видеть.

---

Ты работаешь в репозитории бэкенда Instagram-клона (NestJS + Prisma + Redis + MinIO +
LiveKit), задеплоенного на Render: `https://backend-instagram-kvv4.onrender.com`,
Swagger — `/api/docs-json`.

Фронт (Next.js) уже переведён на этот API: **147 из 170 endpoint'ов вызываются из
живого UI**. Ни один из них не подтверждён живым ответом, потому что БД лежит.
Твоя задача — сделать так, чтобы API работал полностью и по-настоящему.

## Правила работы

1. **Не угадывай — проверяй.** Каждый вывод подкрепляй командой и её выводом
   (`curl`, `prisma`, логи, тест). Если проверить нечем — так и напиши, не выдумывай.
2. **Не ломай существующие контракты.** Фронт уже написан под текущий Swagger. Любое
   изменение формы ответа — только добавлением полей. Если поле всё же надо изменить,
   вынеси это отдельным пунктом в финальный отчёт с пометкой **BREAKING**.
3. **Ответ везде в конверте** `{ data, errors, statusCode }` — как сейчас. Новые
   endpoint'ы должны отдавать так же.
4. **Каждый новый endpoint — в Swagger** (`@ApiOperation`, `@ApiResponse`, DTO
   с `@ApiProperty`). Фронт генерирует типы из `docs-json`: чего нет в схеме, того
   для фронта не существует.
5. **Не коммить секреты.** Значения env — в Render, в репозиторий только
   `.env.example` с именами.
6. **Пиши тесты** на то, что чинишь, если в проекте есть тестовый раннер.
7. Идёшь строго по приоритетам ниже. Задача 1 блокирует всё остальное.
8. **Не отмечай задачу сделанной, пока не выполнен её «Критерий приёмки».**
9. Если чего-то нельзя сделать из кода (нужен доступ к панели Render, платный план,
   чужой сервис) — **не имитируй решение**. Занеси в раздел «Нужен человек» и иди
   дальше.

## Задача 1 🔴 БЛОКЕР: БД, Redis и MinIO недоступны

**Симптом:**

```bash
curl -s https://backend-instagram-kvv4.onrender.com/api/health
{"data":{"status":"degraded","database":"down","redis":"down","storage":"down","uptimeSec":64,...}}
```

**Что уже установлено (проверено 16.07.2026, перепроверять не нужно, но можно):**

| Запрос                                   | Ответ                     | Вывод                           |
| ---------------------------------------- | ------------------------- | ------------------------------- |
| `POST /api/auth/register` с пустым телом | **400** + ошибки по полям | `class-validator` жив           |
| `GET /api/posts` без токена              | **401**                   | Guard'ы живы (они идут до БД)   |
| `POST /api/auth/check-username`          | **500** `INTERNAL_ERROR`  | Любое обращение **к БД** падает |
| `POST /api/auth/login`                   | **500** `INTERNAL_ERROR`  | То же                           |

Само NestJS-приложение работает: 400 и 401 отвечают штатно, падает только то, что
идёт в БД. В момент проверки `uptimeSec` был **64** — процесс только что поднялся
с нуля и **свежий инстанс всё равно не достучался ни до одного из трёх сервисов**.
Перезапуск не лечит.

**Главная гипотеза: в прод уехал конфиг локальной разработки.** В самом Swagger
примеры такие:

```
MusicDto.coverUrl      → "http://localhost:9000/instagram/music/covers/x.webp"   ← MinIO
PostMusicDto.streamUrl → "http://localhost:3000/api/music/35/stream"
LiveTokenDto.wsUrl     → "ws://localhost:7880"                                    ← LiveKit
```

Внутри Render `localhost` — это сам контейнер. То, что **все три** сервиса упали
разом, ложится ровно в эту версию.

**Что сделать:**

1. Найди в коде, как читаются `DATABASE_URL`, `REDIS_URL`, конфиг MinIO/S3 и LiveKit.
   Выпиши **точные имена** всех env-переменных, которые нужны приложению, и покажи,
   где каждая используется.
2. Проверь `.env.example` / `docker-compose.yml` — сверь с тем, что ожидает код.
   Найди расхождения (переменная переименована, а `.env.example` старый, и т.п.).
3. Проверь, что дефолты в коде не подставляют `localhost` молча. Если подставляют —
   **это баг**: на проде отсутствующая переменная должна валить старт с внятной
   ошибкой, а не тихо уводить на `localhost`. Почини: fail-fast на старте
   (валидация конфига через Joi/Zod), с явным списком чего не хватает.
4. Проверь, что `health` показывает **причину**, а не только `down`. Сейчас наружу
   уходит обезличенное `"Internal server error"`, и диагностировать нечем. Добавь
   в `/health` поле с текстом ошибки подключения (без секретов: без пароля и хоста).
5. Убери из Swagger-примеров `localhost` — примеры должны показывать прод-URL.

**Нужен человек (занеси в отчёт, если не можешь сам):** проверить в панели Render
переменные `DATABASE_URL`, `REDIS_URL`, MinIO/S3, LiveKit; открыть логи сервиса —
там настоящая ошибка Prisma (неверный URL, БД suspended, лимит подключений,
IP-allowlist). Если Postgres на free tier (Neon/Supabase) — он засыпает сам, нужен
wake или платный план.

**Критерий приёмки:**

- `GET /api/health` → `"status":"ok"`, `database`/`redis`/`storage` — все `"up"`;
- `POST /api/auth/check-username` → 200/409, **не** 500;
- полный прогон `register → login → GET /auth/me` отвечает 200.

## Задача 2 🔴 Медиа должно отдаваться с публичного хоста

`next/image` на фронте не грузит хост, которого нет в `images.remotePatterns` — это
защита фреймворка. Пока `avatarUrl` приходит как `http://localhost:9000/...`,
картинок в приложении не будет вообще.

**Что сделать:**

1. Найди **все** места, где формируется публичный URL файла (аватары, `media[].url`,
   `coverUrl`, `MusicDto.coverUrl`, `streamUrl`, обложки эфиров/актуальных).
2. Убедись, что URL строится из **одной** переменной (например `PUBLIC_MEDIA_URL`),
   а не склеивается по месту. Если склеивается — вынеси в один хелпер/сервис.
3. Проверь, что в БД не лежат уже сохранённые абсолютные URL с `localhost`. Если
   лежат — **нужна миграция данных**: храни ключ (`key`), а абсолютный URL собирай
   при отдаче. Так смена домена больше никогда не сломает старые записи.
4. Настрой публичный доступ бакета на чтение (или отдавай presigned URL).

**Критерий приёмки:** `avatarUrl` и `media[].url` из живого ответа открываются в
браузере, домен — не `localhost`. Сообщи фронту точный домен: его добавят в
`remotePatterns`.

## Задача 3 🔴 Сокет: нечем авторизоваться и не описаны события

Хендшейк `/socket.io/` отвечает — gateway поднят. Пользоваться им нельзя:

**3.1.** Access-токен у фронта лежит в **httpOnly-cookie** (правильно: из JS он
недоступен, это защита от XSS). Сокет идёт cross-origin, кука туда не уедет, а
достать её из JS фронт не может — и не должен.

**Сделай так:** `POST /socket/ticket` (обычный HTTP, там кука есть) →
`{ ticket: "<одноразовый, TTL 30 сек>" }`. Тикет хранится в Redis, при
`io(url, { auth: { ticket } })` gateway его проверяет, **сжигает** (одноразовый!) и
привязывает сокет к userId. Просроченный или повторно использованный тикет →
`disconnect`.

Альтернатива, если проще: CORS с origin фронта + `withCredentials`, чтобы кука
доехала. Выбери сам, но **напиши в отчёте, что выбрал**.

**3.2.** Ни одного имени события нет в документации. Фронт не будет угадывать имена —
это гарантированно молчащий сокет.

**Сделай так:** документируй все события (Markdown-файл `docs/socket-events.md` в
репозитории бэкенда) — имя, направление (server→client / client→server), payload
с типом. Как минимум эти, если они есть в коде:

```
message:new       server→client  MessageDto
message:read      server→client  { chatId, messageId, userId }
typing            обе стороны    { chatId, userId, isTyping }
live:comment      server→client  LiveCommentDto
live:like         server→client  { liveId, likesCount }
live:viewers      server→client  { liveId, viewersCount }
live:join-request server→client  JoinRequestDto
```

Если каких-то событий в коде **нет** — не выдумывай их: напиши, каких нет, и
реализуй те, что нужны для чата и эфира.

**3.3.** `POST /chats/{id}/call` описан как «WebRTC-сигналинг через сокет», но
сигналинг нигде не описан. Опиши события звонка (offer/answer/ice/hangup) в том же
файле, либо честно напиши в отчёте, что звонки не реализованы на сервере.

**Критерий приёмки:** есть рабочий способ авторизации сокета и файл со всеми
событиями и payload'ами. Приложи пример подключения на `socket.io-client`.

## Задача 4 🔴 LiveKit: `wsUrl` указывает на машину зрителя

`POST /live/start`, `/live/{id}/join`, `/live/requests/{id}/accept` возвращают
`LiveTokenDto` с `wsUrl: "ws://localhost:7880"`. Для браузера `localhost` — это
**компьютер самого зрителя**. Видео не придёт никогда, ни с каким SDK.

**Что сделать:**

1. Вынеси хост LiveKit в env (`LIVEKIT_WS_URL`), отдавай его в `wsUrl`.
2. Это должен быть **`wss://`**, не `ws://`: фронт работает по https, браузер
   заблокирует незащищённый сокет.
3. Подтверди в отчёте, что `token` — обычный LiveKit access token (фронту тогда
   хватит `livekit-client`), и с какими grants он выдаётся: publisher для
   хоста/гостя, subscriber для зрителя.
4. Проверь, что комната реально создаётся при `/live/start` и закрывается при
   `/live/{id}/end`.

**Нужен человек:** поднятый и доступный из интернета LiveKit-сервер (LiveKit Cloud
или свой) + `LIVEKIT_API_KEY` / `LIVEKIT_API_SECRET` в Render. Если сервера нет —
так и напиши, это решение владельца проекта, а не код.

**Критерий приёмки:** `wsUrl` из живого ответа — `wss://`-адрес, который отвечает на
хендшейк.

## Задача 5 🔴 Комментарии в эфире можно отправить, но нельзя прочитать

Есть `POST /live/{id}/comment` → возвращает только что созданный комментарий.
`GET /live/{id}/comments` **не существует**. API умеет только писать. Live-чат без
чужих комментариев — не live-чат.

**Сделай:** `GET /live/{id}/comments?cursor=&limit=` → `LiveCommentDto[]`, курсор —
id последнего элемента (как в остальных списках проекта), сортировка новые→старые,
`limit` по умолчанию 20, максимум 50. Доступ — те же правила, что у `/live/{id}/join`
(блокировка, приватность).

**Критерий приёмки:** два аккаунта, второй пишет комментарий → первый видит его в
`GET /live/{id}/comments`.

## Задача 6 🔴 Хосту не доходит id заявки в эфир

`POST /live/requests/{id}/accept` и `/decline` принимают id заявки. Проследи путь
этого id — он никуда не ведёт:

- `GET /live/{id}/requests` не существует — списка заявок нет;
- `JoinRequestDto` с id возвращается **тому, кто попросился**, а не хосту;
- `NotificationDto` несёт `postId`, `commentId`, `storyId`, `noteId` — и **не несёт**
  `requestId`. У типов `LIVE_JOIN_REQUEST`, `LIVE_JOIN_ACCEPTED`, `LIVE_JOIN_DECLINED`
  нет даже `liveId`.

Из-за этого 2 endpoint'а из 18 физически недостижимы, а split-экран с гостем собрать
не из чего.

**Сделай оба пункта:**

1. `GET /live/{id}/requests?status=PENDING` → `JoinRequestDto[]` — **только хост**,
   иначе 403.
2. Добавь в `NotificationDto` поля `liveId?: string` и `requestId?: number` и
   заполняй их для `LIVE_*` типов. Это заодно чинит `LIVE_STARTED`: сейчас по этому
   уведомлению **некуда перейти**, id эфира в нём нет.

**Критерий приёмки:** хост видит заявку в `GET /live/{id}/requests`, принимает её по
этому id, гость получает publisher-токен.

## Задача 7 ⚠️ Нет «постов по локации»

`PostDto.location` есть, обратного запроса нет: получить посты, снятые в конкретном
месте, нечем. `/search?q=` отдаёт сами локации, но не их ленту. В IG это одна из
основных страниц; сейчас страница локации на фронте пустая.

**Сделай:** `GET /locations/{id}/posts?cursor=&limit=` → `PostDto[]` — ровно как
`/search/hashtag/{name}` (приватные аккаунты и блок исключить так же).

## Задача 8 ⚠️ Курсор не работает там, где у DTO нет `id`

Курсор в проекте — «id последнего элемента предыдущей страницы». У этих DTO своего
`id` нет, поэтому «показать ещё» физически нечем, и фронт берёт разом `limit=50`,
а дальше список обрезается:

- `StoryViewerDto` (`GET /stories/{id}/viewers`)
- `ActivityItemDto` (`GET /profile/me/activity`)
- `ProfileViewDto` (`GET /notifications/profile-views`)

**Сделай:** добавь в эти DTO `id` (id строки БД) — или отдавай `nextCursor` в ответе.
Выбери одно и применяй **единообразно**.

## Задача 9 ⚠️ Нет поиска пользователя по точному `userName`

`@упоминание` ведёт на `/u/{userName}`, а роуты профиля — на uuid. Фронт вынужден
искать через `/users?q=` и сам выбирать точное совпадение, потому что подстрочный
поиск матчит и `fullName` (`er` → и `eraj`, и `america`).

**Сделай:** `GET /users/by-username/{userName}` → `UserBriefDto`, 404 если нет.
Регистронезависимо.

## Задача 10 ⚠️ `GET /auth/me` — неясно, что возвращает

Описан как «текущий пользователь + профиль», но по схеме не видно: голый
`AuthUserDto` или конверт `{ user, profile }`. Фронт сейчас **принимает оба** —
это лишняя ветка, которая живёт только из-за неопределённости.

**Сделай:** зафиксируй один формат, опиши его в Swagger точной схемой (не `object`),
и напиши в отчёте какой. Ничего не меняй в поведении — только опиши как есть.

## Задача 11 ⚠️ Коллекции сохранений односторонние

`POST /posts/{id}/favorite` принимает `collection` (имя), но списка коллекций нет —
прочитать их обратно нельзя. Поэтому в UI ввод имени руками вместо выбора из списка,
как в IG.

**Сделай:** `GET /profile/me/collections` → `{ name, postsCount, coverUrl }[]`
(`coverUrl` — превью последнего поста в коллекции).

## Задача 12 ⚠️ В уведомлении нет превью поста

В `NotificationDto` есть `postId`, но нет картинки, а в IG в строке уведомления
справа стоит миниатюра поста.

**Сделай:** добавь `postThumbUrl?: string` — URL первого медиа поста.

## Задача 12.1 ⚠️ Нет «постов по треку»

`MusicDto.usesCount` говорит «использован в 42 публикациях» — связь в БД есть, а запроса,
который эти публикации отдаст, нет. В IG страница трека — это лента постов с этим звуком.
Та же дыра, что и задача 7 (посты по локации): счётчик есть, списка нет.

**Сделай:** `GET /music/{id}/posts?cursor=&limit=` → `PostDto[]` — как
`/search/hashtag/{name}` (приватные аккаунты и блок исключить так же).

## Задача 13: полный прогон и отчёт

Когда задачи 1–12 закрыты:

1. Прогони **весь** happy path вживую, curl'ом, и приложи вывод:
   `register → login → /auth/me → создать пост → лента → лайк → коммент → сторис →
чат → эфир`.
2. **Сверь каждый ответ с тем, что обещает Swagger.** Расхождения между схемой и
   реальностью — это баг, и чинить надо **схему или код**, а не молчать. Прецедент:
   на прошлом бэкенде `gender` отдавался строкой `"Male"`, а на запись принимал
   только `0|1` — фронт узнал об этом только вживую. Ищи именно такое:
   enum'ы, nullable-поля, отсутствующие `id`, разные типы на чтение и на запись.
3. Заведи тестовый аккаунт с наполненными данными (несколько постов, сторис, чат)
   и отдай логин/пароль — фронту нужно на чём проверять.
